android-View的探索-三View,Window,Activity之间的关系

发表于 2017-03-13 | 分类于 Android源码研究 | | 阅读次数

日常开发中我们接触到的主要是View以及Activity，并且在此基础上开发，但是其实View跟Window是离不开的，所以本文旨在理清View,Window以及Activity之间的关系。

在Activity中，我们通过setContentView()方法将View与Activity进行挂钩起来,关于setContentView()的源码分析可以看我写的文章， 这里就不详细在做介绍，要找到View,Window,Activity三者之间的关系，我们需要从Activity实例化的地方开始进行看起，当我们startActivity()的时候会最终回调到ActivityThread的名为H的handler类中，在类中会调用到handleLaunchActivity(...)，在handleLaunchActivity(...)中会调用performLaunchActivity(..)来获取一个新的Activity实例，那么主要研究的目标就在于当前方法中：

在performLaunchActivity（...）中主要就是对Activity的实例化，然后调用onCreate(..)方法，其中在上述步骤中有个activity.attach(...)方法中会新建一个Window对象用来存放Activity的各种View：

在代码中其实可以看见新建的Window对象是一个PhoneWindow对象，其实在setContentView()的源码分析的分析当中已经知道，settContentView()方法最终是调用了PhoneWindow中的setContentView()方法进行设置,其中的步骤是：

  • 启动一个DecorView(本质是FrameLayout)对象
  • 调用LayoutInflater进行解析xml文件，解析完成后添加到DecorView当中

每一个Activity的视图在新建的时候会同时创建一个ViewRootImpl，即一个Window对应一个ViewRoot对应一个View,在ViewRootImpl中会将View通过root.setView(view, wparams, panelParentView)方法设置进去，那么主要的方法就在setView(…)中:

在requestLayout()中，系统会执行scheduleTraversals()方法，scheduleTraversals()是绘制View的实际入口，那么绘制完成之后继续调用mWindowSession.addToDisplay()来通知底层系统进行WIndow的添加的过程，mWindowSession是一个Binder对象，类型是IWindowSession,其在ViewRootImpl初始化的时候通过mWindowSession = WindowManagerGlobal.getWindowSession()来获取得到，获取到的是一个Session类，我们里面查询到daaToDisplay()方法：

在代码里面可以看到，其实就是通过WindowManagerService来进行处理通知底层进行window的添加的操作。

上面的一大段代码解释了Window创建以及View的添加过程，简而言之，就是我们View添加进了Window中，window通过调用addView(…)方法经过各种会调用最终调用IWindowSession实现类Session的addToDisplay(…)方法，在addToDisplay(…)方法中通知了WMS来处理窗口添加的过程。

总结：

我们在初始化一个根view(可以理解为setContentView())时，其实将view添加到window中，借由window来进行通知交互的一个过程，一个根View对应一个window对应一个ViewRoot，ViewRoot是页面绘制的核心类，如我们在view中measure(),layout(),onDraw()方法皆由当前类为媒介通知系统进行对应的工作。
